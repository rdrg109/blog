1 Some terminology

If you don't know what a sexp is. I encourage you to read the "Intro eval" section of the Elisp manual. Here's the relevant part that defines the term "sexp".

A Lisp object that is intended for evaluation is called a “form” or “expression”(1). The fact that forms are data objects and not merely text is one of the fundamental differences between Lisp-like languages and typical programming languages. Any object can be evaluated, but in practice only numbers, symbols, lists and strings are evaluated very often.

-----–— Footnotes -----–—

(1) It is sometimes also referred to as an “S-expression” or “sexp”, but we generally do not use this terminology in this manual.

2 The problem

Let's suppose you want to make all #+BEGIN_SRC code blocks whose language is text to contain a timestamp. Some users might think that this can be accomplished by executing the following. 

(setq org-babel-default-header-args:text
      '((:prologue . "(format-time-string \"%m/%d/%Y %H:%M:%S %Z\")")))

The problem with this is that the sexp wouldn't be evaluated when the text code block is expanded. That is, you would end up with the string "(format-time-string \"%m/%d/%Y %H:%M:%S %Z\")" as the prologue of the code block.

Some other users might think that this can be accomplished by executing the following. 

(setq org-babel-default-header-args:text
      `((:prologue . ,(format-time-string "%m/%d/%Y %H:%M:%S %Z"))))

The problem with this is that the expression would be evaluated once and this would occur when evaluating the setq expression. For this reason, this doesn't do what we are looking for.

3 The solution

You can get the desired behavior by modifying the value of the org-keyword-properties variable.

This variable is the one that is automatically updated with the values of the #+PROPERTY affiliated keyword whenever an Org Mode file is opened. 

(setq org-keyword-properties
      '(("HEADER-ARGS:TEXT" . ":prologue (format-time-string \"%m/%d/%Y %H:%M:%S %Z\")")))

Of course, this needs more tinkering since if you use setq on that variable, you would lose the previous content of the variable. Recall that this variable holds all the properties that has been set through the #+PROPERTY affiliated keyword and is locally set whenever an Org Mode file is opened, so an appropiate modification of that variable is needed.
